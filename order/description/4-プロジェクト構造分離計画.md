# プロジェクト構造分離計画

## 現状の問題

現在、環境構築用のスクリプトとアプリケーションで使用するスクリプトが混在している。

## 分離方針

### 環境構築用（setup/）
- データベース初期化
- 開発環境セットアップ
- テスト・検証用スクリプト

### アプリケーション用（app/ または src/）
- アプリケーションで使用するコード
- ビジネスロジック
- 共通ユーティリティ

## 提案する構造

```
PostgreSQL/
├── setup/                          # 環境構築用
│   ├── init-scripts/              # データベース初期化スクリプト
│   │   ├── 01-init.sql
│   │   ├── 02-tables.sql
│   │   ├── 03-indexes.sql
│   │   ├── 04-seed-data.sql
│   │   ├── 05-vector-migration.sql
│   │   ├── 06-vector-search-examples.sql
│   │   └── 07-add-embeddings.sql
│   ├── scripts/                   # 環境構築用スクリプト
│   │   ├── setup-venv.ps1
│   │   ├── setup-venv.sh
│   │   ├── setup.sh
│   │   ├── load-env.ps1
│   │   └── run-example.ps1        # テスト・検証用
│   └── examples/                  # サンプル・検証用スクリプト
│       ├── e5-usage-example.py
│       └── calculate-similarity-example.py
│
├── app/                            # アプリケーション用（新規作成）

│   │   ├── embedding_service.py   # ベクトル生成サービス

│       ├── generate-embeddings.py

│
├── docker-compose.yml              # 環境構築用
├── env.template                    # 環境構築用
├── requirements.txt                # 依存関係（環境構築用とアプリ用が混在）
├── requirements-app.txt            # アプリケーション用依存関係（新規作成）
├── requirements-setup.txt          # 環境構築用依存関係（新規作成）
└── README.md
```

## メリット

1. **明確な責任分離**: 環境構築とアプリケーションの責務が明確になる
2. **保守性向上**: 各ディレクトリの目的が明確になり、保守しやすくなる
3. **デプロイ時の明確化**: アプリケーション用のコードが明確になる
4. **依存関係の明確化**: 環境構築用とアプリ用の依存関係を分離できる
